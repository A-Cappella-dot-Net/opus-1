# presto

The messaging middleware of opus-1. Presto provides fault-tolerant publish/subscribe with snapshot semantics — the layer that lets application components exchange live data, bootstrap their state, and fail over between instances. Presto itself is transport independent — it defines the messaging logic without committing to a wire. Like cembalo, it builds directly on [continuo](../continuo/README.md) (pooled objects, pipe/sink sockets). [Presto-aeron](../presto-aeron/README.md) supplies the transport-dependent side for Aeron. Bindings for other transports could be implemented alongside it in the same fashion.

- **Publish/subscribe with snapshots** (`ps`) — the SnS ("snap and subscribe") protocol: a new subscriber receives a snapshot of current state from a cache or snapshot provider, merged seamlessly with subsequent live publications. Pluggable merge managers (`PassThru`, `KeyBased`, `Queued`) decide how the two streams reconcile; a high-water mark and timeout bound the snapshot phase. `PrestoClient` / `PubSubClient` are the client-side entry points.
- **Content filtering** (`ps.sql`) — subscriptions are expressed with SQL-like syntax; `SqlParser` and the `WhereNode` tree evaluate where-clauses against published objects, so subscribers receive only matching data.
- **Fault-tolerant collective** (`ft.collective`) — daemons form a collective: core members detect each other's liveness over pipe/sink connection pairs and elect a primary, either by first-alive order or by voted quorum. `FtManager` then assigns ACTIVE/INACTIVE status across application groups (with a configurable number of actives per group), notifying members and monitors as instances come and go. All state transitions run on a single event thread.
- **Staleness monitoring** (`monitor`) — pluggable predicates and actions detect when data or instances go stale and react accordingly.
- **Serializer** (`serializer`) — the transport-independent side of the serializer, the component that stamps explicitly serialized messages with global sequence numbers so their consumers observe the same total order. The Aeron implementation lives in presto-aeron; the deployable service in the `serializer` module.
